executive summary

The best path is evolutionary.

Flue's strongest reusable ideas are not its branding or server architecture. They are its clean contracts around agents, tools, sandboxes, connectors, and schema-backed results. workflow can adopt those ideas while keeping its current identity: deterministic orchestration over coding harnesses.

how workflow works now

Current architecture strengths and gaps.

strength

Deterministic workflow scripts

A run walks a JS/TS script top-to-bottom. Each agent() call receives a sequence number, which makes journal replay simple and defensible.

strength

Crash-safe journal replay

Cached sequence entries return without respawning models. This is the core differentiator: restart and resume are cheap, observable, and understandable.

strength

Harness adapters

Claude, Codex, Copilot, and raw API are represented behind one runner shape, with typed capabilities for schemas, token reports, and tool events.

strength

Event-stream UI

The runtime emits a small set of events that reduce into run state. This keeps CLI, UI, logs, and replay aligned around one observable surface.

gap

Per-call options repeat intent

Scripts currently express model, adapter, isolation, label, and schema on individual calls. That is explicit, but larger workflows will repeat the same agent personality.

gap

Extension contracts are implicit

The repo has adapters and worktree isolation, but there is not yet a crisp public contract for third-party sandboxes or agent-installable integrations.

Decision constraint: any borrowed idea should preserve deterministic sequencing, journal replay, and the event model. If an idea makes replay ambiguous, it belongs outside the core runtime.

Agent profiles

Named presets for repeated agent behavior.

Each agent() call carries options directly. This is simple, but repetition grows quickly in real workflows.

Add profile() or exported profile objects that define adapter, model, instructions, isolation, default schema behavior, and labels.

1

Highest value with low architectural risk.

Sandbox connector spec

Provider-neutral remote execution contract.

workflow has worktree isolation and process runners, but no documented contract for remote sandboxes like Daytona, E2B, Modal, or in-house runners.

Define a small interface for readFile, writeFile, stat, readdir, exec, timeout, env, cwd, and cancellation.

2

Unlocks remote execution without hardcoding providers.

Structured result protocol

Normalize schema capture across adapters.

workflow converts Zod to JSON Schema and validates returned data. Adapter quality still depends on harness-specific schema behavior.

Introduce a tool-backed finish / give_up protocol for adapters that support tools, with validation feedback and retry.

3

Improves reliability, especially for weaker schema adapters.

Agent-installable connectors

Markdown instructions as integration packages.

Integrations are code and docs. A user must infer where to place provider-specific glue.

Add workflow add sandbox daytona or workflow add adapter raw-api that prints exact instructions an agent can apply.

4

Good distribution pattern after the contract exists.

Persistent sessions

Follow-up interactions after a run.

workflow is centered on finite runs. Watching, resuming, and saving all point at run IDs.

Optionally add sessions as a layer around a saved run or workflow, but keep them separate from deterministic run replay.

5

Useful, but only after core extension points are clear.

Build targets

Generate deployable artifacts.

The CLI runs, watches, resumes, stops, saves, and lists. It does not package workflows as deployable apps.

Later: workflow build --target node|github-actions|container for teams that want scheduled or hosted workflow execution.

6

Valuable, but premature if execution contracts are still moving.

what the first improvement could look like

Agent profiles without losing explicitness.

today
phase("Review")
const findings = await agent(prompt, {
  adapter: "claude",
  model: "sonnet",
  isolation: "worktree",
  schema: FINDINGS,
  label: "review-api",
})
possible
export const reviewer = profile({
  adapter: "claude",
  model: "sonnet",
  isolation: "worktree",
  instructions: "Review for correctness risks only.",
})

phase("Review")
const findings = await agent(reviewer, prompt, {
  schema: FINDINGS,
  label: "review-api",
})
Why this is safe: profiles are static inputs to the same deterministic agent calls. The sequence number, journal key, semaphore behavior, schema validation, and event stream do not need to change. The runtime only resolves defaults before constructing the existing request.
recommended roadmap

Small slices, each independently useful.

01

Introduce profiles as data

Add types and tests first. Profiles should be mergeable defaults, not new runtime control flow.

02

Write the sandbox connector spec before code

Define the external contract in docs. Then implement one local/reference connector to prove the interface.

03

Normalize structured results behind adapters

Keep native schema support where it is good. Add tool-backed finish only where the adapter can support it cleanly.

04

Add connector installation docs

Once contracts settle, make workflow add emit exact agent-readable instructions and file templates.

05

Revisit sessions and deployment targets

Only add long-lived agents or build targets when user workflows clearly need hosted, repeated, or external-triggered execution.

decision guide

How to choose what to build.

Question Build if yes Defer if
Does it preserve deterministic replay? Profiles, connector specs, structured result protocols. It introduces hidden background work that changes the order or count of agent calls.
Does it reduce workflow author friction? Profiles and installable connector guides. It mainly adds platform surface before scripts are easier to write.
Does it make adapters more reliable? Schema finish tools, sandbox timeout/cancellation contracts. It depends on provider-specific behavior that cannot be tested locally.
Does it clarify the public extension boundary? Sandbox connector spec and reference implementations. It requires committing to deployment targets before execution APIs are stable.
Recommended decision: build agent profiles first, then a sandbox connector contract. Treat Flue as validation that these extension surfaces matter, not as a blueprint for a full rewrite.
